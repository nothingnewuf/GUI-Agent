# GUI Agent 算法设计说明文档

## 1. 整体架构

本项目 Agent 基于多模态大模型（Doubao-Seed-1.6-vision），采用单轮决策架构：每次仅根据当前截图和历史上下文，输出一个操作动作（CLICK / TYPE / SCROLL / OPEN / COMPLETE）

核心流程如下：

```
用户指令 + 当前截图 → [System Prompt] → [历史上下文] → 多模态模型 → 动作解析 → 标准化输出
```

Agent 继承自 `BaseAgent`，重写了 `generate_messages()` 和 `act()` 方法，内部维护多轮对话历史，通过设计的 System Prompt 和多级解析策略，将模型的自然语言输出转化为标准化的动作指令

## 2. System Prompt 设计

### 2.1 核心原则

System Prompt 强调三个核心原则：

- **当前截图优先**：历史截图仅用于理解操作上下文，决策严格基于标注为"当前步骤"的截图，避免历史信息干扰
- **坐标必须精确**：明确归一化坐标系的定义（左上角 (0,0)，右下角 (1000,1000)），提供常见 UI 元素的参考坐标区间（状态栏、标题栏、导航栏等）
- **一步一动作**：限制每次只输出一个动作，避免多步预测带来的误差累积

### 2.2 搜索流程专项指引

针对最高频的搜索类任务，在 Prompt 中明确三步流程：

1. **状态判断**：先观察截图判断搜索框是否已激活（键盘是否弹出、输入框是否有光标闪烁）
2. **激活搜索框**：若未激活则 CLICK 搜索框/搜索图标
3. **输入内容**：确认激活后才 TYPE

提供"口诀级"判断标准：能看到键盘弹出 → TYPE；看不到键盘 → 先 CLICK 搜索入口，有效减少 CLICK/TYPE 误判

### 2.3 坐标参考体系

在 Prompt 中提供详细的坐标估算方法和关键 UI 元素参考点：

- 比例估算法：元素在屏幕宽度 1/4 处则 x≈250，右侧 2/3 则 x≈667
- 常见参考点：左上角返回键 [40, 75]、右上角搜索 [900, 75]、底部导航各标签位置
- 区域划分：状态栏 y:0-50、标题栏 y:50-150、内容区 y:150-900、导航栏 y:900-1000

### 2.4 多场景覆盖

Prompt 覆盖常见场景的处理策略：广告/弹窗关闭、返回操作、列表选择、筛选按钮定位、任务完成判断等，减少模型在边界情况下的决策失误

## 3. 历史管理策略

### 3.1 截图保留策略

在对话历史中保留最近 3 轮的实际截图（而非纯文本描述），使模型能够回顾之前的界面状态，理解操作序列的上下文关系

### 3.2 历史标签机制

每条历史消息使用明确的 `[历史步骤 N]` 标签进行标注，与当前步骤的 `[当前步骤]` 标签形成清晰对比，帮助模型区分"历史参考"和"当前决策依据"。历史截图配合文本描述（Thought 摘要 + 动作记录），兼顾视觉上下文与 token 效率

### 3.3 窗口管理

历史窗口控制在最近 3 轮（6 条消息），超出部分自动截断，在提供足够上下文的同时，避免过长历史导致的注意力分散和 token 消耗过大。

## 4. 多级动作解析策略

模型输出的原始文本格式可能不稳定，采用三级降级解析策略：

### 4.1 结构化解析（第一级）

使用正则表达式匹配标准格式：

```
Action: CLICK
Point: [x, y]
```

支持中英文冒号、多种参数名变体（Point/point/坐标）、带/不带方括号的坐标格式。坐标值自动 clamp 到 [0, 1000] 范围内。SCROLL 动作同时支持 StartPoint/EndPoint 及中文变体

### 4.2 JSON 降级解析（第二级）

匹配 ` ```json ``` ` 代码块或裸 JSON 对象，解析 `action` 和对应参数字段，支持字段名变体（如 point/coordinate/coords、text/content、app_name/app）

### 4.3 宽松兜底解析（第三级）

处理完全不规范的输出格式：

- 兼容 BaseAgent 默认格式 `click(point='<point>x y</point>')`
- 兼容函数调用格式 `open(app_name='应用名')`、`type(content='文本')`
- 最后尝试从文本中提取动作关键词 + 任意坐标对
- 避免将 "not complete" 等否定表述误判为 COMPLETE
- 对 COMPLETE 动作额外检查，排除否定语境

三级策略确保在任何可理解的模型输出下都能提取有效动作，不会因格式问题导致任务中断

## 5. 自适应修正

### 5.1 应用名模糊修正

**问题**：模型偶尔输出应用名的简写形式（如"去哪旅行"），与参考数据中的完整官方名称（"去哪儿旅行"）不匹配。由于评测框架做精确匹配，简写会导致 OPEN 动作判定失败

**方案**：不依赖硬编码的映射表，而是基于编辑距离算法计算模型输出与已知应用名列表的字符串相似度。只有当相似度超过 0.85 且唯一匹配时才进行修正，避免歧义误判

实例：

- "去哪旅行" → "去哪儿旅行"（相似度 0.96）—— 修正
- "去哪" → 低于阈值 0.53 —— 不修正

## 6. 工程优化

### 6.1 Token 控制

- max_tokens 设为 512，在保证 Thought 完整性的前提下限制输出长度
- temperature 设为 0，确保模型输出确定性
- 历史截图窗口限制为 3 轮，平衡上下文与 token 消耗
- 历史中 Thought 截断至 100 字符，减少冗余

### 6.2 错误处理

API 调用失败时返回 COMPLETE 动作，避免评测进程因异常中断，同时记录详细错误日志便于调试

### 6.3 日志记录

每个步骤记录模型原始输出、解析结果、token 使用量，便于问题定位和性能分析

## 7. 测试结果

在 11 个测试用例上，测试结果：

- 总用例数: 11
- 通过用例数: 8
- 用例准确率: 72.73%
- 步骤准确率: 91.43%

## 8. 适用场景与局限性

**适用场景**：本方案适用于基于多模态大模型的移动端 GUI 自动化任务，特别是搜索、导航、媒体播放等常见移动应用操作流程

**已知局限性**：

- 部分复杂多步骤任务，如航班搜索、外卖下单的场景，存在坐标精度不足的问题：模型在归一化坐标系下对目标元素的中心点估算存在偏差，尤其在搜索框和查询按钮的定位上容易出现系统性误差
- 搜索框激活状态判断不稳定，模型在部分截图中对键盘弹出、光标闪烁等激活标志的识别存在误判，导致 CLICK/TYPE 动作类型选择错误
- 任务完成时机判断缺乏一致性，模型在处理多步骤任务时，有时过早输出 COMPLETE（未完成全部子任务），有时在已完成后又继续操作
- 历史截图会带来信息干扰，虽然提供了上下文，但也可能分散模型对当前截图的注意力
- 不同 App 的 UI 布局差异大，固定的坐标参考点无法覆盖所有界面风格
